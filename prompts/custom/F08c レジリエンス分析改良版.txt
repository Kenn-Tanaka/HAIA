# レジリエンス分析プロンプト Ver.2.2
## このプロンプトの目的

事故・インシデントを分析する際、通常は
「誰が悪かったか」「どこでミスしたか」を探しがちです。

しかし、このプロンプトは**別の問い**を立てます：

> **「前提が崩れたとき、システムはどこまで持ちこたえられたか？」**

---

## 【重要】分析にあたっての入力制約（必ず守ること）

* この分析は、**提供された資料・記録・ログのみに依拠**する
* 資料に明示されていない事柄を【事実】として断定してはならない
* 判断できない場合は、無理に補わず **「不明」** と記述する
* もっともらしさよりも、**検証可能性・出典の明確さ**を優先する

---

## 基本原則（分析を始める前に読んでください）

### ❶ 責任追及が目的ではありません

* 「誰が悪いか」を探す分析は禁止
* 「なぜ破局を防げなかった構造になっていたか」を探ります

### ❷ 人間の間違いは「通常運転」です

* 判断ミス・ルール逸脱・勘違いは**起こる前提**
* 問題は、それが即破局につながる**設計・運用構造**です

### ❸ 時間軸を最優先します

* いつ気づけたか
* どれくらい猶予があったか
* なぜその時間窓が使われなかったか

---

## 分析の5ステップ

---

## 問1：何が想定外だったか？

### 書くべきこと

事故の背景には、
**「これは起こらない」「考えなくてよい」**とされていた前提があります。

それを **3〜5個**、箇条書きで列挙してください。

各前提について、以下を書くこと：

---

### ① その前提は何だったか？

（具体的・運用レベルで）

### ② なぜそう思われていたか？

（過去実績・規程・慣行など）

### ③ 実際には何が起きたか？

（前提がどう崩れたか）

### ④ 前提が崩れた背景に、日常運用の圧力・最適化はなかったか？

（時間・手間・顧客影響・定時性など）

**④を書く際の注意点：**
* 単なる推測で終わらせず、**資料から読み取れる運用上の制約条件**を根拠にすること
* 可能であれば、時系列データから**前提が崩れていた期間の長さ**を明示すること
* 例：「【事実】○月○日から△月△日まで約X日間、アカウントが放置されていた」→「【推定】この期間、定期的な棚卸しプロセスが機能していなかったと考えられる」

---

### 📌 記述ルール：タグの厳格運用

すべての記述の冒頭に、以下のタグを付けてください。

| タグ       | 定義                                |
| -------- | --------------------------------- |
| **【事実】** | 一次資料に同趣旨の明示的記述があり、出典（p/節/時刻等）を示せる |
| **【推定】** | 直前に列挙した【事実】から合理的に導ける解釈            |
| **【仮説】** | 可能性の一つに過ぎず、現時点では裏付けがない            |

#### 追加ルール（重要）

* 【事実】には必ず出典を付ける
* 【推定】を書く場合、**根拠となる【事実】を最低1つ併記**する
* 【仮説】を書く場合、**「反証可能性（確認すべき追加データ）」を1行添える**
* 心理状態・バイアス語（正常性バイアス等）は、証拠なしに【事実】として書かない

**推定の質を高めるルール：**
* 【推定】を書く際は、**「事実に基づく推定」**であることを明示する
* 例：「【事実に基づく推定】2025-05-04から侵入まで約5ヶ月間、アカウントが放置されていた【事実】ことから、定期的な棚卸しプロセスが機能していなかったと考えられる」

---

## 問2：システムはどう反応したか？

想定外が起きたとき、
**システム（人＋機械＋組織）はどう振る舞ったか**を、時間軸で評価します。

---

### ステップ1：時間を区切る

出来事を **3〜5フェーズ** に分けてください。

```
フェーズ1：異常の初期検知（時刻）
フェーズ2：専門家介入（時刻〜）
フェーズ3：引き継ぎ・判断継続（時刻〜）
フェーズ4：停止・収束
```

---

### ステップ2：各フェーズを評価する

以下の表を埋めてください。

| 評価項目        | 評価内容                  |
| ----------- | --------------------- |
| **システムの反応** | 柔軟だったか／硬直していたか（具体行動で） |
| **被害の広がり**  | 局所→拡大のどこにいたか          |
| **介入の余地**   | 時間窓はあったか／なぜ使われなかったか   |
| **検知シグナル**  | 本来見えるべきだった異常・前提崩壊の兆候  |
| **根拠（出典）**  | 報告書p/節/時刻・ログ・証言       |
| **確度**      | 高／中／低                 |

※ 評価語だけで終わらせず、**必ず具体的行動・判断を書くこと**

**介入の余地を評価する際の追加観点：**
* 時間窓が存在した場合、**その長さを定量的に示す**こと
  * 例：「介入の余地：極めて長い時間窓（約5ヶ月＝150日）があったが、全く利用されなかった」
* 時間窓が使われなかった理由を、以下の観点で分析すること：
  * **検知の失敗**：シグナルが見えなかった／見過ごされた
  * **判断の失敗**：シグナルは見えたが重要性を認識できなかった
  * **対応の失敗**：重要性は認識したが対応手段がなかった／優先順位が低かった

---

## 問3：段階的に悪化を防げなかったか？

### ❶ どこが「All or Nothing」だったか

* 成功 or 失敗の二択になっていた判断・設計はどこか
* 中間状態を許さなかった箇所はどこか

**❶を書く際の注意点：**
* 単に「二択だった」と指摘するだけでなく、**欠落していた中間層を具体的に列挙**すること

**記述例（分野別）：**

#### サイバーセキュリティの例

**認証システム**:
- 【事実】単一要素認証のみで、突破されれば即座に侵入を許す設計。
- 【欠落していた中間層】
  - 接続元制限（IPアドレス、地理的位置）
  - 多要素認証
  - 異常パターン検知・遮断

#### 化学プラントの例

**圧力制御システム**:
- 【事実】設定圧力を超えたら即座に緊急停止する設計。
- 【欠落していた中間層】
  - 段階的な圧力リリーフ（低圧弁→中圧弁→高圧弁）
  - 警報による操作員への通知と手動介入の余地
  - 原料供給の段階的削減

#### 医療の例

**投薬プロセス**:
- 【事実】処方指示をそのまま実行する設計。
- 【欠落していた中間層】
  - 薬剤師による処方監査
  - 投与前の患者確認（複数人によるダブルチェック）
  - 投与量の段階的増量プロトコル

---

### ❷ どこで劣化・縮退させられたか

```
【通常運転】→【要注意／制限モード】→【完全停止】
                  ↑
              ここで止められた
```

---

### ❸ 縮退モードの仕様書を書く（必須）

**基本フォーマット：**

```
【縮退モード名】：○○モード

1. 発動トリガー（何を検知したら発動するか）
2. 制限内容（通常運転と比べて何を制限するか）
3. 通知・ログ（誰に何を伝えるか）
4. 解除条件（どうなったら通常に戻すか）
5. フォールバック（制限による業務・プロセスへの影響をどう緩和するか）
6. 誤作動・濫用を防ぐガードレール
```

※ 数値・条件を書く場合は「なぜその設定か」を一言添えること

**❳を書く際の追加要件：**

各項目について、以下の詳細を含めること：

**1. 発動トリガー：**
* 定量的な閾値を示す（例：「短時間での3回以上の失敗」）
* なぜその閾値か、根拠を一言添える

**3. 通知・ログ：**
* アラートの重要度分類（高/中/低）を明示
* エスカレーション基準を含める
  * 例：「3件以上の高重要度アラートが1時間以内に発生した場合、経営層へエスカレーション」

**4. 解除条件：**
* 解除前の確認事項を具体的に列挙
  * 脅威の完全な排除確認
  * 影響範囲の特定
  * 再発防止策の実施確認
* 解除権限者を明確化（役職名まで）

**5. フォールバック：**
* 業務継続性への配慮を明示
* 代替手段の提供プロセスを記述

**6. 誤作動・濫用を防ぐガードレール：**
* 誤検知時の業務影響度評価方法を含める
* 監査ログの保存期間と確認頻度を明示

---

**記述例（分野別）：**

#### サイバーセキュリティの例

```
【縮退モード名】：異常アクセス制限モード

1. 発動トリガー
   - 短時間（10分以内）に3回以上のログイン失敗
   - 通常と異なる国・地域からのアクセス試行

2. 制限内容
   - 該当アカウントの一時ロック（30分間）
   - アクセス元IPアドレスのブロック（24時間）

3. 通知・ログ
   - セキュリティ担当者への即時アラート（重要度：高）
   - 該当ユーザーへのメール通知
   - 全イベントをSIEMに記録
   - 3件以上/1時間でCISOへエスカレーション

4. 解除条件
   - セキュリティ担当者による手動確認・解除
   - または30分経過後の自動解除（IPブロックは継続）
   - 解除権限者：セキュリティ責任者以上

5. フォールバック
   - 代替認証手段の提供（電話認証等）
   - ヘルプデスクでの本人確認後の手動解除

6. ガードレール
   - 管理者アカウントは自動ロック対象外
   - 解除履歴の監査ログ保存（1年間）
   - 同一アカウントで1日3回以上の制限発動は別途調査
```

#### 化学プラントの例

```
【縮退モード名】：反応器高温警報運転モード

1. 発動トリガー
   - 反応器温度が設定値+10℃を超過
   - または温度上昇率が5℃/分を超過
   （根拠：過去の運転データから、この値を超えると暴走反応のリスクが高まる）

2. 制限内容
   - 原料供給量を通常の50%に自動削減
   - 冷却水流量を最大に自動増加
   - 攪拌速度を維持（停止させない）

3. 通知・ログ
   - 中央制御室への警報（音声＋表示、重要度：高）
   - プラントマネージャーへの自動通報
   - DCS（分散制御システム）への記録
   - 30分以内に正常化しない場合、工場長へエスカレーション

4. 解除条件
   - 温度が設定値+5℃以下に低下
   - かつ、操作員が状況確認・原因特定を完了
   - プラントマネージャーの承認
   - 解除権限者：プラントマネージャー以上

5. フォールバック
   - 製品品質は低下する可能性があるが安全を優先
   - 削減した供給量分は次バッチで調整
   - 納期への影響は生産管理部門と調整

6. ガードレール
   - 原料供給は完全停止しない（ポンプシール保護のため）
   - 温度が設定値+20℃を超えたら緊急停止に移行
   - 手動での制限解除は2名以上の承認を要する
   - 解除後24時間は温度監視を5分間隔に強化
```

#### 医療の例

```
【縮退モード名】：患者急変時診療制限モード

1. 発動トリガー
   - バイタルサインの急激な変化（血圧30mmHg以上低下、SpO2 90%以下等）
   - または患者からのナースコール（緊急ボタン）
   （根拠：早期警告スコア（EWS）が基準値を超過）

2. 制限内容
   - 該当病室への新規入室制限
   - 予定されていた検査・処置の延期
   - 応援スタッフの招集

3. 通知・ログ
   - 担当医・看護師への緊急呼び出し（重要度：最高）
   - 病棟責任者への通知
   - 電子カルテへの自動記録
   - 5分以内に医師到着なしの場合、RRS（院内迅速対応システム）起動

4. 解除条件
   - 患者状態の安定確認（バイタルサイン正常化）
   - 担当医による解除判断
   - 必要に応じて集中治療室等への転室完了
   - 解除権限者：担当医以上

5. フォールバック
   - 延期した検査・処置の再スケジュール
   - 他病棟への応援要請
   - 患者家族への状況説明

6. ガードレール
   - 他の患者の安全確保（最低限の巡回は継続）
   - 制限時間の上限設定（2時間以上は病棟責任者の再判断）
   - 誤報の場合も記録を残し、トリガー設定の妥当性を定期評価
```

---

### ❹ 根本的な設計変更案

* 運用改善ではなく **仕組み自体を変える案**
* **最小実装（〜1年）／拡張（〜3年）**の2段で記述してもよい
* 既存技術で実現可能な範囲を意識する

**記述時の注意：**
* 各提案の**優先順位と根拠**を明示する
* **概算コストと期待効果**を記述する（定性的でも可）
* **実装ロードマップ**を示す（四半期単位推奨）

**ただし、以下の点に留意：**
* コスト・ロードマップは**資料に根拠がある場合のみ**記述
* 推定で書く場合は【推定】タグを付け、「一般的な○○分野では」等と注釈を加える
* 分野によって時間軸が大きく異なることを認識する
  * サイバー：数ヶ月〜1年
  * 化学プラント：設計変更は数年、定修時の改造は1〜2年
  * 医療：手順変更は数ヶ月、システム変更は1〜2年

**推奨フォーマット（柔軟に適用）：**

```
【最小実装（短期）】
- 提案1（優先度：最高／高／中、効果：高／中／低、コスト：高／中／低）
- 提案2...

【拡張（中長期）】
- 提案3...
```

**コスト・便益分析について：**
* 資料に損失額やコスト情報がある場合は記述を推奨
* ない場合は「【推定】一般的な○○業界では△△程度のコストが想定される」程度に留める

## 問4：何が足りなかったか？

### レジリエンスの4能力（よ・み・こ・ま）

| 能力     | 意味            |
| ------ | ------------- |
| **予測** | 将来のリスクを見通す    |
| **監視** | 今何が起きているか把握する |
| **対応** | 異常に素早く対処する    |
| **学習** | 過去から教訓を得る     |

### 書き方

```
【欠けていた能力】：○○

・どこに欠落があったか：
・なぜ欠けていたか：
```

※ **3〜5個まで**に限定すること

**問4の追加要件：**

**1. 能力間の相互関係を明示すること：**

各能力の欠落が、次の能力の欠落を引き起こす悪循環を分析する。

```
【能力間の相互作用】

予測の欠如 → 監視対象の不足 → 対応の遅延 → 学習機会の喪失
という負のサイクルが存在した。

具体例：
- 一時アカウントの残存リスクを予測できない
  ↓
- アカウント棚卸しを監視項目に含めない
  ↓
- 不正利用を検知できず対応できない
  ↓
- インシデント後も根本原因を学習できない
```

**2. 組織的要因の分析を追加すること：**

技術的な能力だけでなく、以下の組織的要因も分析する：

```
【欠けていた能力】：組織的ガバナンス

・どこに欠落があったか：
  - セキュリティ投資の意思決定プロセスの不備
  - IT部門のリソース不足（人員・予算）
  - 経営層のセキュリティ意識の不足
  - 部門間の責任分界の曖昧さ

・なぜ欠けていたか：
  - 【推定】過去に大きなインシデントがなく、危機感が醸成されていなかった
  - 【推定】セキュリティ投資が「コスト」と捉えられ、優先順位が低かった
  - 【推定】IT部門とセキュリティ部門の役割分担が不明確だった
```

## 問5：他の事例にも使える教訓は？

### 書き方ルール

* 固有名詞・業界依存語を避ける
* 適用できる分野を書く
* **適用できない条件（反例）を1行書く**

### テンプレート

```
【教訓○】：〜〜〜

■ 内容：
■ 適用できる分野：
■ 適用条件：
■ 適用できない条件（反例）：
```

**問5の追加要件：**

**1. 優先順位マトリクスの作成：**

各教訓について、以下の4軸で評価し、表形式で整理すること：

| 教訓 | 効果 | コスト | 実装期間 | 優先度 |
|------|------|--------|----------|--------|
| 教訓1 | 高/中/低 | 高/中/低 | X-Yヶ月 | 最優先/高/中/低 |

**2. 実装の難易度と前提条件を明示：**

各教訓について、以下を追加記述する：
* 実装の難易度（技術的/組織的）
* 成功のための前提条件
* よくある失敗パターンとその回避策

**記述例：**

```
【教訓3】：「認証」は「多層防御」で臨む。

■ 内容：
  IDとパスワードによる認証だけでなく、多要素認証（MFA）や生体認証など、
  複数の認証要素を組み合わせることで、認証情報の漏洩による
  不正アクセスのリスクを大幅に低減する。

■ 適用できる分野：
  VPN接続、リモートアクセス、クラウドサービスへのログイン、
  重要なシステムへのアクセス

■ 適用条件：
  ユーザーが複数の認証要素（例：パスワード、SMSコード、認証アプリ、指紋）を
  保持・利用できる環境であること

■ 適用できない条件（反例）：
  単一の認証要素（例：パスワードのみ）でしか認証できないシステム

■ 実装の難易度：
  - 技術的：低〜中（既存のMFAソリューションが豊富）
  - 組織的：中（ユーザー教育とサポート体制の構築が必要）

■ 成功のための前提条件：
  - 経営層の承認とリソース配分
  - ユーザー向けの丁寧な説明とトレーニング
  - ヘルプデスクの準備

■ よくある失敗パターン：
  - ユーザーの反発を恐れて導入を躊躇する
    → 段階的導入（まず管理者から）で実績を作る
  - MFAの利便性が低く、利用率が上がらない
    → 使いやすいソリューションの選定とUXの改善
```

## 付録：用語集

### 基本概念（全分野共通）

| 用語 | 意味 | 例 |
|------|------|-----|
| **制御ループ** | 「観測→判断→行動」のサイクル | 温度を測る→高いと判断→冷却する |
| **フィードバック** | 行動の結果が制御者に返ってくる情報 | 冷却後の温度を再測定 |
| **インターロック** | ある条件を満たさないと次の動作ができない仕組み | 扉が閉まらないと機械が動かない |
| **フェイルセーフ** | 故障しても安全側に倒れる設計 | 停電したら信号機は全て赤になる |
| **縮退運転** | 一部機能を制限して安全を確保する | 速度制限運転、出力制限運転 |

### 分野別用語

#### 🔐 サイバーセキュリティ

| 用語 | 意味 | 例 |
|------|------|-----|
| **多要素認証（MFA）** | 2つ以上の認証要素を組み合わせる認証方式 | パスワード + SMSコード |
| **ゼロトラスト** | 「決して信頼せず、常に検証する」セキュリティモデル | 社内ネットワークでも全てのアクセスを検証 |
| **EDR/XDR** | エンドポイント／拡張検知・対応システム | 端末の振る舞いを監視して脅威を検知 |
| **SIEM** | セキュリティ情報・イベント管理システム | 複数のログを統合分析 |
| **セグメンテーション** | ネットワークを複数の区画に分割する設計 | DMZ、社内LAN、サーバーセグメントの分離 |

#### 🏭 化学プラント・プロセス産業

| 用語 | 意味 | 例 |
|------|------|-----|
| **リリーフ弁（安全弁）** | 圧力が設定値を超えたら自動的に開く弁 | 反応器の過圧防止 |
| **トリップ** | 異常検知時の自動緊急停止 | 高温トリップ、高圧トリップ |
| **DCS** | 分散制御システム | プラント全体を監視・制御するシステム |
| **バッチプロセス** | 一定量ずつ処理する方式（連続プロセスの対） | 医薬品製造、食品加工 |
| **暴走反応** | 制御不能な発熱反応の加速 | 冷却失敗による温度・圧力の急上昇 |

#### 🏥 医療・ヘルスケア

| 用語 | 意味 | 例 |
|------|------|-----|
| **バイタルサイン** | 生命徴候（体温、血圧、脈拍、呼吸等） | 患者状態の基本的指標 |
| **ダブルチェック** | 2人以上で独立に確認する手順 | 投薬前の薬剤・投与量確認 |
| **タイムアウト** | 処置前に全員で患者・部位等を確認する手順 | 手術開始前の最終確認 |
| **RRS** | 院内迅速対応システム（Rapid Response System） | 急変の兆候がある患者への早期介入 |
| **EWS** | 早期警告スコア（Early Warning Score） | バイタルサインから患者の状態悪化を予測 |

#### 🚆 運輸・交通

| 用語 | 意味 | 例 |
|------|------|-----|
| **ATS/ATC** | 自動列車停止／制御装置 | 信号超過や速度超過時の自動ブレーキ |
| **デッドマン装置** | 運転士の意識喪失を検知する仕組み | 一定時間操作がないと警報→自動停止 |
| **インシデント** | 事故には至らなかったが危険だった事象 | ヒヤリハット |
| **フールプルーフ** | 人間の誤操作を防ぐ設計 | 2つのボタンを同時に押さないと動作しない |

#### ⚡ 電力・エネルギー

| 用語 | 意味 | 例 |
|------|------|-----|
| **負荷遮断** | 需給バランス維持のため一部負荷を切り離す | 計画停電、自動負荷制限 |
| **周波数維持** | 電力系統の安定性指標 | 50Hz/60Hzからの逸脱監視 |
| **ブラックスタート** | 全停電後の電源再起動手順 | 外部電源なしでの発電再開 |
| **N-1基準** | 任意の1設備が停止しても系統全体は運転継続できる設計 | 送電線1本が故障しても停電しない |
